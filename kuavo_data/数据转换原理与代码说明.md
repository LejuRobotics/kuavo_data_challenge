# Kuavo 数据转换原理与代码逐行说明

本文档说明 **rosbag → Lerobot parquet** 的转换原理，并逐段解释 `kuavo_data` 模块中的关键代码。

---

## 一、整体原理

### 1.1 目标

- **输入**：Kuavo 机器人录制的 ROS bag（多话题、多相机、不同时间戳）。
- **输出**：Lerobot 格式的 parquet 数据集（按固定频率对齐的「帧」，每帧包含 observation + action）。

### 1.2 核心问题

- 各话题（关节、相机、夹爪等）**发布频率不同**，时间戳不对齐。
- 若一次性把整个 bag 读入内存再对齐，**内存会爆**（尤其多相机+深度图）。

### 1.3 两阶段流式方案（本仓库做法）

| 阶段 | 做什么 | 内存占用 |
|------|--------|----------|
| **第一阶段** | 只遍历 bag，**只读时间戳**，不读图像/大消息 | 几 MB（时间戳列表） |
| **第二阶段** | 按「主时间线」分块，每块只读该时间范围内的消息 → 对齐 → 写入 dataset → 释放 | 每块可控（由 chunk_size 决定） |

这样**不会**把整个 bag 同时放进内存，适合大 bag 和有限内存的机器。

### 1.4 主时间线（main timeline）

- 以**某一相机**的帧率为基准（通常选帧数最多的相机），得到一串「目标时间戳」。
- 所有其他话题（关节、夹爪、其他相机等）都**按时间戳最近邻**对齐到这些目标时间戳上。
- 输出帧数 = 主时间线在「丢弃首尾 + 降采样」后的长度。

### 1.5 数据流概览

```
rosbag 目录
    → list_bag_files() 得到多个 .bag
    → 对每个 bag：
        → [Phase 1] scan_timestamps_only() → main_timestamps, all_timestamps
        → [Phase 2] process_in_chunks():
            → 按 chunk 读消息 → _align_single_frame() 得到 aligned_frame
            → frame_callback(on_frame) 把对齐后的帧转成 Lerobot 的 state/action/images
            → add_frame() 写入 LeRobotDataset
            → save_callback 里 save_episode() + create_hf_dataset()，再清 buffer、gc
    → 得到 lerobot/ 下的 parquet + 视频等
```

下面按文件、按函数逐段说明代码在做什么。

## 二、入口：`CvtRosbag2Lerobot.py`

### 2.1 文件头与补丁（约 1–38 行）

```python
import lerobot_patches.custom_patches  # 必须最先导入，扩展 Lerobot 的 FeatureType 等
import os, gc, shutil
from pathlib import Path
from typing import Literal
import numpy as np, torch, tqdm, hydra
from omegaconf import DictConfig
from lerobot.datasets.lerobot_dataset import HF_LEROBOT_HOME as LEROBOT_HOME
from lerobot.datasets.lerobot_dataset import LeRobotDataset
import dataclasses
from kuavo_data.common import kuavo_dataset as kuavo
```

- **lerobot_patches**：给 Lerobot 打补丁，例如支持 RGB+Depth、自定义归一化等，**必须**在其它 lerobot 使用前导入。
- **LeRobotDataset**：Lerobot 的数据集类，最终写入的 parquet/video 都由它管理。

### 2.2 `DatasetConfig` 与 `create_empty_dataset_chunked`（约 56–171 行）

**DatasetConfig**：控制 Lerobot 数据集创建方式（是否用视频、时间容差、写图像进程数等）。

**create_empty_dataset_chunked** 的作用：**在本地创建空的 LeRobotDataset**，并声明好「有哪些 observation / action 键、维度、相机」等，为后续 `add_frame` 做准备。

- **motors / action_dim / state_dim**：由 `DEFAULT_JOINT_NAMES_LIST` 和 `kuavo` 的配置决定（手臂+夹爪/灵巧手等）。
- **ONLY_HALF_UP_BODY**：若为 False，会多加 base 相关维度（cmd_pos_x/y/yaw、ctrl_change_cmd 等）。
- **features**：为每个 observation（state、各相机 RGB、各相机 depth）和 action 定义 `dtype`、`shape`、`names`，Lerobot 据此建表。
- 深度相机在 features 里用 `observation.{cam}`（如 `observation.depth_h`），RGB 用 `observation.images.{cam}`。
- 若本地已存在同 `repo_id` 的目录会先 `shutil.rmtree`，再 `LeRobotDataset.create(...)`，得到空数据集。

### 2.3 `populate_dataset_chunked`（约 174–331 行）

这是**真正按 chunk 流式写入**的逻辑。

- **参数**：`dataset`（空 LeRobotDataset）、`bag_files`、`task`、`episodes`（要处理的 episode 下标）、`chunk_size`（每块帧数）。
- **bag_reader = kuavo.KuavoRosbagReader()**：封装了「要读哪些话题、用什么函数解析每条消息」的映射。
- **frames_buffer**：当前 episode 内，积累的「已对齐、已转成 Lerobot 格式」的帧列表；每满一个 chunk 或 episode 结束就写盘并清空。

**on_frame(aligned_frame, frame_idx)**（每帧回调）：

- **aligned_frame**：来自 `chunk_process` 的「单帧对齐结果」，键为话题 key（如 `observation.state`、`action`、`observation.images.head_cam_h`、`action.kuavo_arm_traj` 等），值为 `{"data": ..., "timestamp": ...}` 或类似。
- **get_array(key, dtype)**：从 `aligned_frame` 里安全取数组，没有则返回空数组或 None。
- **state / action**：先取原始 state 和 action 数组；再用 **arm_traj**（或 alt）填到 `action[12:26]`（手臂部分）；若 state/action 为空则直接 return，不写入。
- **手部**：根据末端类型读 `claw` / `qiangnao` / `rq2f85` 的 state 和 action；若都为空则 return。
- **手部归一化**：二值化（IS_BINARY）或按比例缩放到 [0,1]（如 /100、/0.8、/255）。
- **final_state / final_action**：按 `which_arm`（left/right/both）用 `get_hand_slice` 拼出左/右臂+手部的 state 和 action，再 `np.concatenate` 成最终向量。
- **ONLY_HALF_UP_BODY == False** 时：追加 `cmd_pos_world` 和 `gap_flag`（轨迹断点用 999 标记）到 action。
- **构建 frame 字典**：  
  - `observation.state`、`action`：torch 张量。  
  - 每个相机：若有 depth，做 clip + 归一化到 [0,255] 再 repeat 成 3 通道；否则直接 `observation.images.{cam}`。  
  - 若有 velocity/effort 也放进 frame。  
- 最后 **frames_buffer.append(frame)**，frame_count 加 1。

**on_chunk_done()**（每个 chunk 结束回调）：

- 若 buffer 非空：给每帧加上 `task`，**dataset.add_frame(frame)**，然后 **dataset.save_episode()**、**dataset.hf_dataset = dataset.create_hf_dataset()**，再 **frames_buffer.clear()**、**gc.collect()**，实现「写一块、放一块」的内存控制。

**process_rosbag_chunked**：

- 内部会先 **Phase 1：scan_timestamps_only**，再 **Phase 2：process_in_chunks**（见下一节）。
- 每个 chunk 处理完会调用 **save_callback**，这里传入的就是 **on_chunk_done**。
- Episode 末尾若还有未满 chunk 的帧，后面会再统一 **add_frame + save_episode + create_hf_dataset** 一次（约 406–413 行）。

失败 bag 会记入 **failed_bags**，最后写入 `error.txt`。

### 2.4 `port_kuavo_rosbag_chunked`（约 334–379 行）

- 若已存在同 `repo_id` 的目录则删掉。
- **bag_reader.list_bag_files(raw_dir)**：得到该目录下所有 `.bag` 并排序。
- **n**：若指定，则随机抽 n 个 bag 用于转换（否则全用）。
- **create_empty_dataset_chunked** 建空数据集，**populate_dataset_chunked** 往里填数据；**chunk_size** 从参数传入（默认 100）。

### 2.5 `main` 与 Hydra 配置（约 383–435 行）

- **@hydra.main**：从 `../configs/data/` 读 `KuavoRosbag2Lerobot` 配置。
- **kuavo.init_parameters(cfg)**：根据 yaml 把 `only_arm`、`eef_type`、`which_arm`、相机列表、TRAIN_HZ、RESIZE 等全部灌进 `kuavo_*` 模块的全局变量，供后续 `create_empty_dataset_chunked` 和 `on_frame` 使用。
- **raw_dir / version / lerobot_dir**：  
  - `raw_dir = cfg.rosbag.rosbag_dir`，  
  - `version = cfg.rosbag.lerobot_dir`，  
  - `lerobot_dir = os.path.join(raw_dir, "../", version, "lerobot")`，即输出到「与 rosbag 目录同级」的 `version/lerobot`。
- **DEFAULT_JOINT_NAMES_LIST**：根据 `USE_LEJU_CLAW` / `USE_QIANGNAO` 和 `which_arm` 拼出「手臂+夹爪/灵巧手」的关节名列表，和 config 里的 slice 一致（UP_START_INDEX=13 等），用于 **create_empty_dataset_chunked** 的 features。
- 最后调用 **port_kuavo_rosbag_chunked(raw_dir=..., repo_id=..., task=..., root=lerobot_dir, n=n, chunk_size=chunk_size)**，完成整次转换。

## 三、分块处理核心：`chunk_process.py`

### 3.1 `ChunkedRosbagProcessor` 的职责

- 只做两件事：  
  1）**scan_timestamps_only**：第一遍扫 bag，只拿时间戳；  
  2）**process_in_chunks**：第二遍按时间窗口分块读、对齐、回调。

- 不关心「state/action 怎么从消息解析」，那由 **msg_processer** 和 **topic_process_map** 提供（在 kuavo_dataset 里定义）。

### 3.2 `scan_timestamps_only(bag_file)`（约 42–110 行）

- **all_timestamps**：字典，key 为「逻辑 key」（如 `observation.state`、`head_cam_h`），value 为该话题所有消息的 **时间戳列表**（不读消息体，所以省内存）。
- **main_timeline**：在配置的 **camera_names** 里，选**帧数最多**的相机作为主时间线；输出时长由它决定。
- **jump = main_timeline_fps // train_hz**：主相机 30Hz，train_hz=10 则 jump=3，即每 3 帧取 1 帧。
- **main_timestamps**：对主相机时间戳先 **丢弃首尾 sample_drop 帧**，再按 jump 降采样，得到「目标时间戳序列」。
- **返回值**：main_timeline（str）、main_timestamps（list）、all_timestamps（dict）。后面 Phase 2 用 main_timestamps 做「每帧对齐」的时间轴，用 all_timestamps 做各话题的索引。

### 3.3 `process_in_chunks(...)`（约 112–208 行）

- 再次打开 bag，**按时间范围**读消息（start_time / end_time），只读当前 chunk 需要的区间，避免全量加载。
- **timestamp_arrays**：把 all_timestamps 转成 numpy 数组，便于二分查找。
- **_precompute_alignment_indices**：对每个主时间戳、每个话题，用**二分**找「最近的一条消息」的索引，得到 alignment_indices[key][global_idx]。
- **_detect_arm_traj_gaps**：若启用下肢等，会检测 `action.kuavo_arm_traj` 时间戳是否出现大间隔；间隔内用 999 填充，和主分支逻辑一致。
- **对每个 chunk**：  
  - 取该 chunk 的 main_timestamps 子序列；  
  - 用 **chunk_start_time / chunk_end_time**（前后各加一点 margin）调用 **_read_chunk_data**，只读这段区间内的消息到 **chunk_data**；  
  - 对 chunk 内每个 (global_idx, main_stamp)，调用 **_align_single_frame** 得到 **aligned_frame**，再 **frame_callback(aligned_frame, global_idx)**；  
  - 然后 **del chunk_data**，调用 **save_callback()**，便于上层写盘和 gc。

### 3.4 `_read_chunk_data(bag, start_time, end_time)`（约 259–307 行）

- 用 **bag.read_messages(..., start_time=..., end_time=...)** 只读该时间范围内的消息（若 ROS 支持）；否则回退到全量读再按时间过滤。
- 对每条消息用 **topic_process_map[key]["msg_process_fn"](msg)** 解析成 `{"data": ..., "timestamp": ...}`，存入 **chunk_data[key][timestamp]**。
- 返回结构：**{ topic_key : { timestamp : msg_data } }**，供 _align_single_frame 按时间取用。

### 3.5 `_align_single_frame(...)`（约 309–363 行）

- **aligned_frame** 初始只有 `timestamp: main_stamp`。
- 对 **topic_process_map** 里每个 key：  
  - **kuavo_arm_traj 间隙**：若当前 main_stamp 落在已检测的 gap 内，则该 key 填 999，不查 chunk_data。  
  - 否则用 **alignment_indices[key][global_idx]** 得到该话题「最近邻」的索引，进而得到 **target_ts**；再在 **chunk_data[key]** 里找时间最接近 target_ts 的那条消息，写入 **aligned_frame[key]**。  
- 这样每一帧都是「以主时间线为轴、其他话题按时间最近邻对齐」的一帧，交给上层的 **on_frame** 转成 Lerobot 的 state/action/images。

### 3.6 `_precompute_alignment_indices`（约 210–254 行）

- 对每个话题的时间戳数组，对每个 main_timestamps 里的 stamp 做 **bisect_left**，再比较前一个/当前谁离 stamp 更近，得到最近邻索引列表，避免在 process_in_chunks 里每帧都二分一遍。

### 3.7 `_load_bag`（约 365–374 行）

- 若 bag 未索引会先尝试 **reindex_rosbag**，再打开；失败则 **allow_unindexed=True** 打开。

## 四、话题映射与消息解析：`kuavo_dataset.py`

### 4.1 `init_parameters(cfg)`（约 46–89 行）

- 从 **config_dataset.load_config(cfg)** 得到 Config 对象，把 **DEFAULT_CAMERA_NAMES、TRAIN_HZ、MAIN_TIMELINE_FPS、SAMPLE_DROP、SLICE_*、USE_DEPTH、RESIZE_*、ONLY_HALF_UP_BODY、USE_LEJU_CLAW、USE_QIANGNAO、TASK_DESCRIPTION** 等全部写入模块全局变量，供 **KuavoMsgProcesser**、**KuavoRosbagReader**、**CvtRosbag2Lerobot** 使用。

### 4.2 `KuavoMsgProcesser`（约 92–318 行）

每个方法都是「ROS 消息 → 统一格式」：返回 **{"data": np.array 或 image, "timestamp": float}**。

- **process_color_image**：支持 CompressedImage 和原始 Image；BGR→RGB，再 **resize 到 (RESIZE_W, RESIZE_H)**。
- **process_depth_image**：从 compressedDepth 里解 PNG，得到 uint16 深度图，再 resize。
- **process_joint_state**：从 sensors_data_raw 取 joint_q。
- **process_kuavo_arm_traj**：position 转弧度。
- **process_cmd_pos_world**：从 twist 取 linear.x/y、angular.z。
- **process_claw_state / process_claw_cmd**：夹爪状态/指令。
- **process_rq2f85_* / process_qiangnao_* / process_dex_state**：仿真夹爪、灵巧手等。
- **process_sensors_data_raw_extract_arm**：从 joint 里只取手臂 14 维 [13:27]。

这些函数在 **KuavoRosbagReader._topic_process_map** 里被挂到对应话题上。

### 4.3 `KuavoRosbagReader`（约 321–457 行）

- **_topic_process_map**：  
  - key：逻辑名（如 `observation.state`、`action`、`action.kuavo_arm_traj`、`observation.images.head_cam_h`、`depth_h` 等）；  
  - value：`{"topic": "/ros/topic/name", "msg_process_fn": ...}`。  
- 相机名来自 **DEFAULT_CAMERA_NAMES**（由 config 的 which_arm、use_depth 决定），分别映射到 `/cam_h/...`、`/cam_l/...`、`/cam_r/...` 的 color 和 depth 话题。
- **process_rosbag_chunked**：  
  - 构造 **ChunkedRosbagProcessor**（传入 msg_processer、topic_process_map、camera_names、train_hz、main_timeline_fps、sample_drop、only_half_up_body）；  
  - 先 **scan_timestamps_only(bag_file)**；  
  - 再 **process_in_chunks(..., frame_callback, chunk_size, save_callback)**。  
- **list_bag_files(bag_dir)**：`glob.glob(os.path.join(bag_dir, '*.bag'))` 并排序。
- **align_frame_data**：旧版「全量读入再对齐」的逻辑，用于非 chunked 的 process_rosbag；chunked 路径不用这里。

## 五、配置加载：`config_dataset.py`

### 5.1 `Config` 与 `load_config`（约 6–159 行）

- **Config**：only_arm、eef_type、which_arm、use_depth、depth_range、train_hz、main_timeline、main_timeline_fps、sample_drop、is_binary、resize、task_description 等。
- **use_leju_claw / use_qiangnao / only_half_up_body**：由 eef_type、only_arm 推导。
- **default_camera_names**：根据 which_arm 和 use_depth 返回头/腕/深度相机的列表。
- **slice_robot / dex_slice / claw_slice**：手臂、灵巧手、夹爪在「全关节/全手」数组中的切片，供 CvtRosbag2Lerobot 里拼 **final_state / final_action** 时使用。
- **load_config(cfg)**：从 Hydra/OmegaConf 的 cfg 里按路径取出 yaml 中的 dataset 字段，填到 Config，返回实例。

## 六、小结：数据转换流程一览

1. **配置**：Hydra 读 yaml → **init_parameters(cfg)** 填满 kuavo 全局变量和 Config。
2. **输出目录**：根据 rosbag 目录和 lerobot_dir 得到 **lerobot_dir**，删除已存在的同 repo 数据。
3. **关节名与 features**：根据 which_arm、eef_type 得到 **DEFAULT_JOINT_NAMES_LIST** 和 features → **create_empty_dataset_chunked** 建空 LeRobotDataset。
4. **每个 bag 文件**：  
   - **Phase 1**：**scan_timestamps_only** → main_timeline、main_timestamps、all_timestamps。  
   - **Phase 2**：**process_in_chunks**，按 chunk 读消息 → **_align_single_frame** 得到 aligned_frame → **on_frame** 里解析 state/action/手部/图像 → 拼成 Lerobot 的 frame → **add_frame**；每 chunk 结束 **on_chunk_done** → save_episode、create_hf_dataset、清 buffer、gc。
5. **结果**：在 **lerobot_dir** 下得到 Lerobot 标准格式的 parquet 与视频，可直接给 **kuavo_train** 做模仿学习。

整条链路的核心就是：**两遍扫描 + 主时间线对齐 + 分块读/写/释放**，在保证「多话题时间对齐」的前提下，把内存占用压到可接受范围。
